# 项目开发方法论 (Project Development Methodology)

本技巧涵盖了识别适合 LLM 处理的任务、设计有效的项目架构以及利用助手辅助开发进行快速迭代的原则。该方法论适用于构建批处理流水线、多助手研究系统或交互式助手应用。

## 何时激活

在以下情况下启用此技能：
- 启动一个可能受益于 LLM 处理的新项目
- 评估任务是更适合助手还是适合传统的硬编码
- 为 LLM 驱动的应用设计架构
- 规划一个带结构化输出的批处理流水线
- 在单助手（Single-agent）与多助手（Multi-agent）方案间做选择
- 估算 LLM 重型项目的成本和时间线

## 核心概念

### 1. 识别任务-模型适配度 (Task-Model Fit)
并非每个问题都适合用 LLM 解决。第一步是评估任务特征是否与 LLM 的长处对齐。
- **适合 LLM 的任务**：跨源综合信息、基于量表的主观判断、自然语言输出、对错误有一定的容忍度、批处理任务。
- **不适合 LLM 的任务**：精确计算（数学、计数）、实时性要求极高（亚秒级响应）、完美准确性要求（Hallucination 风险）、由于数据漂移导致模型缺乏必要的背景知识。

### 2. 手动原型验证 (The Manual Prototype Step)
在投入自动化开发前，先手动测试。将一个具有代表性的输入复制到模型界面，观察输出质量。如果手动原型失败，自动化系统也必将失败。
- 验证模型是否具备相关知识。
- 验证模型是否能产出所需格式。
- 为后续的提示词设计提供模板。

### 3. 流水线架构 (Pipeline Architecture)
LLM 项目受益于分阶段的流水线架构：
`获取 (Acquire) → 准备 (Prepare) → 处理 (Process) → 解析 (Parse) → 渲染 (Render)`

- 其中 **“处理 (Process)”**（LLM 调用）是非确定性且昂贵的，而其他阶段是确定性的。
- 这种解法允许仅在必要时运行昂贵的 LLM 阶段，而在解析和渲染阶段进行快速迭代。

### 4. 文件系统即状态机 (File System as State Machine)
使用文件系统而非数据库来跟踪流水线状态。每个处理单元分配一个目录，每个阶段的完成通过特定文件（如 `raw.json`, `prompt.md`, `response.md`, `parsed.json`）的存在来标记。
- **优点**：天然的幂等性、易于调试（人类可读）、简单的并行化和缓存。

### 5. 结构化输出设计
提示词设计直接决定了解析的可靠性。提示词应：
- 包含明确的分隔标示（Section markers）。
- 提供格式示例。
- 阐明动机（告知模型输出将被程序解析）。
- 使用受限的值（枚举、分值范围）。

## 架构选择：单助手 vs 多助手

- **单助手流水线**：适用于批处理、任务间无互动、追求低成本和低复杂度的情况。
- **多助手架构**：适用于需要从不同维度并行探索、任务超出单一上下文容量的情况。**核心目的是上下文隔离**。

## 架构减法 (Architectural Reduction)

从极简架构开始。只有在证明有必要时才增加复杂度。生产实践表明，移除过度专门化的工具往往能提高性能。
- **示例 (Vercel)**：将 17 个专门工具减少为 2 个原语（Bash 和 SQL），成功率从 80% 提升至 100%。

## Guidelines

1. 在构建自动化之前，先通过手动原型验证任务与模型的适配度。
2. 将流水线构建为离散的、幂等的、可缓存的任务阶段。
3. 使用文件系统进行状态管理和调试。
4. 设计提示词以产出结构化、可解析的输出，并提供明确示例。
5. 从极简架构开始，仅在必要时增加复杂度。
6. 早期估算成本并在开发过程中持续跟踪。
7. 构建鲁棒的解析器（Parser），以应对 LLM 输出的轻微波动。
