# 贡献指南：上下文工程助手技能

感谢您有兴趣为上下文工程助手技能（Agent Skills for Context Engineering）集合做出贡献。本文件提供了贡献的指南和说明。

## 如何贡献

### 报告问题

如果您发现错误、解释不清楚或缺失主题，请开一个 Issue，并提供：
- 对问题的清晰描述
- 发现问题的技能和章节
- 如果有建议的改进方案，请一并提供

### 提交更改

对于实质性的更改，请：

1. Fork 本仓库
2. 为您的更改创建一个特性分支（feature branch）
3. 按照技能模板结构进行修改
4. 确保 SKILL.md 文件保持在 500 行以内
5. 视情况添加参考资料或脚本
6. 提交 Pull Request，并在描述中清晰说明更改内容

### 添加新技能

添加新技能时：

1. 使用 `template/SKILL.md` 中的模板
2. 遵循命名规范（使用连字符的的小写字母）
3. 包含 SKILL.md 以及相应的参考资料/脚本
4. 更新根目录下的 README.md 以包含新技能
5. 确保内容是平台无关的（适用于 Cursor, Claude Code 等）

## 技能结构要求

每个技能必须包含：

- 包含 `name` 和 `description` 字段的 YAML frontmatter
- 逻辑清晰的章节
- 适当时的实际案例
- 链接到相关技能的集成说明

可选添加项：

- 包含额外文档的 `references/` 目录
- 包含可执行示例的 `scripts/` 目录
- 复杂技能可包含多个 Markdown 文件

## 内容指南

### 写作风格

- 言简意赅，表达精准
- 恰当使用技术术语
- 提供具体的指导，而非模糊的建议
- 提供具体的案例
- 指出复杂性和权衡点

### 避免平台特定性

技能应能在不同的助手平台间通用。请避免：
- 未经抽象的平台特定工具名称
- 供应商锁定的示例
- 仅属于特定助手产品的功能

### 保持技能聚焦

每个技能应只有一个核心。如果一个主题变得过于庞大，请考虑将其拆分为具有清晰依赖关系的多个技能。

## 行为准则

本项目遵循专业的技术协作模式。请尊重不同的观点，并专注于改进集体知识库。

## 问题咨询

关于贡献的任何问题，请开一个 Issue 进行讨论。
